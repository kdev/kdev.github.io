在一个变更中，系统提供了“添加工作指令”功能，可以给一个选定的变更添加任意数量的子任务（即工作指令）。

工作指令的属性

一旦创建了一个变更，这个变更实际上就是一个定义子任务（工作指令）的容器，可以由变更创建人来定义这些工作指令。

表 8.8.  工作指令的属性

属性 必填字段 详情
Title标题 是 简短描述 / 工作指令名称
用法说明 否 工作指令的详细文本描述
工作指令类型 是 定义工作指令类型的选择列表，如审批、工作指令、PIR（实施后审查）等等。
类别 是 定义变更的类型，例如：“3-正常”。
影响 是 定义变更的效果或影响，如“4-高”。
优先级 是 定义这个变更的优先级，例如“5-非常高”。
状态 是 在创建一个新变更地，自动设置变更状态。在修改已有变更时变更创建人和变更经理可以手动设置变更状态。可用的状态和下一状态通过集成的状态模式转换定义，参阅“ 变更状态模式转换 ”一节。
客户请求日期 否 如果需要的话，这个属性可以通过“ChangeEdit编辑变更”、“ChangeAdd添加变更”和“ChangeZoom变更详情”模板的系统配置不激活。这个属性显示客户期望的实施日期。
附件 否 允许附加相关的文件和文档。

除了这些属性之外，工作指令分配的服务人员还可以通过“报告”功能收集属性，详细解释请参阅“工作指令服务人员报告”章节。

工作指令类型

下面的“工作指令类型”属性条目是在OTRS::ITSM变更模块标准安装之后就可用的。

  • 审批

  • 工作指令

  • 撤销

  • 决定

  • PIR (实施后审查)

系统管理员可以自由定义和指定工作指令类型，还可以添加新的类型。

定义工作指令的服务人员

每个工作指令可以分配给一个“工作指令服务人员”，如负责执行这个工作指令的人员。

系统提供了访问所有连接的服务人员后端。重点注意事项：只有完全访问“itsm-change变更管理”组权限的成员才会显示出来。

分配的工作指令服务人员可以将相关信息如注释、实际工作指令开始和结束时间、需要的处理时间和状态整理成文档提交。

链接工作指令到配置项/服务

OTRS::ITSM允许在一个工作指令中链接到另外的对象。

在标准安装中，一个工作指令可以链接到下面的对象：

  • 配置项

    • 配置项类-计算机

    • 配置项类-硬件

    • 配置项类-位置

    • 配置项类-网络

    • 配置项类-软件

  • 服务

  • 工单

这些设置是通过系统配置“Framework ->Core::LinkObject”定义的，可以根据需要扩展。

将一个工作指令保存为模板

通过工作指令本身的“模板”功能可以将一个工作指令保存为模板。

对话框中需要变更创建人指定模板名称和注释(可选)。一旦用户点击“添加”按钮离开对话框，这个工作指令就可以在创建新的工作指令时作为模板使用。
